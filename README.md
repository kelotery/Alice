# Alice(6.17)
# Marvis 复刻项目前端阶段总结：新建对话与自动任务模块

本次开发中，我主要完成了 Marvis 复刻项目前端部分的两个核心板块：**新建对话模块**和**自动任务模块**。这两个模块是整个桌面端 AI 助手的基础入口，一个负责用户与系统的自然语言交互，另一个负责任务的创建、展示和管理。相比单纯堆页面，我这次更关注的是前端整体框架、页面结构、模块之间的关系，以及后续如何继续接入真实的 Agent 能力。

---

## 一、整体前端框架设计思路

这次前端采用的是桌面端应用的设计思路，而不是普通网页页面。整体结构围绕一个主窗口展开，左侧是固定的导航栏，右侧是不同功能页面的内容区。这样的设计比较接近 Marvis 官方客户端的使用体验，也更适合后续扩展为一个本地 AI 助手。

目前前端整体可以理解为：

```text
主窗口
├── 左侧 Sidebar 导航栏
│   ├── 新建对话
│   ├── 自动任务
│   ├── 技能广场
│   ├── 应用
│   ├── 文档
│   ├── 图库
│   ├── 此电脑
│   └── 设置
│
└── 右侧主内容区域
    ├── 新建对话页面
    ├── 自动任务页面
    └── 其他功能页面
```

我这次的重点不是把所有页面都做完整，而是先把最核心的两个功能入口搭出来：**用户从哪里输入任务，以及任务创建后在哪里管理**。因此，“新建对话”和“自动任务”实际上构成了当前项目前端的主流程闭环。

---

## 二、新建对话模块总结

### 1. 模块定位

新建对话模块是整个项目的主入口。它并不是一个简单的聊天页面，而是一个面向 AI 助手的任务输入入口。用户可以在这里通过自然语言输入需求，例如让系统打开应用、查找文件、创建提醒任务，或者执行某个本地操作。

所以我在设计这个模块时，没有把它只当成普通聊天窗口，而是把它定位为：

```text
自然语言输入入口
→ 任务理解入口
→ Agent 调度入口
→ 结果展示入口
```

也就是说，用户在这里输入一句话，后续可以交给 Main Agent 或其他功能模块去判断任务类型，然后再分发给文件、应用、自动任务、系统信息等不同能力。

---

### 2. 页面结构设计

新建对话页面目前主要由以下几部分组成：

```text
新建对话页面
├── 顶部或中央欢迎区域
├── 主输入框
├── 快捷任务提示
├── 执行状态展示
├── 结果展示区域
└── 日志 / 状态反馈区域
```

其中最核心的是输入框和任务反馈区域。输入框负责接收用户的自然语言任务，反馈区域则用于展示执行结果、状态变化或任务日志。

我在设计时尽量避免把页面做成普通聊天软件那种纯消息列表，而是更偏向“AI 操作台”的风格。因为 Marvis 这类产品的重点不是陪聊，而是让用户通过一句话驱动本地任务。

---

### 3. 前端制作思路

新建对话模块的制作思路主要有三点。

第一，我先保证页面结构清晰。左侧导航栏固定，右侧内容区域根据点击切换页面，这样整个项目不会变成零散页面，而是有统一的应用框架。

第二，我把输入区域作为页面视觉中心。因为用户打开软件后，最直接的操作就是输入任务，所以输入框的位置、大小和层级需要比较突出。

第三，我预留了结果展示和状态展示区域。后续无论是接入本地模型、Function Calling、MCP 工具，还是接入文件检索、应用启动、自动任务创建，都可以把结果统一返回到这个页面中展示。

目前新建对话模块已经基本完成了“用户输入任务 → 前端接收 → 展示执行反馈”的基础框架。后续需要继续加强的是多轮对话体验、历史记录、快捷 tag 交互，以及与后端 Agent 调度逻辑的深度连接。

---

## 三、自动任务模块总结

### 1. 模块定位

自动任务模块是本次开发中另一个重点。它的作用是让用户创建、查看和管理定时任务。相比新建对话模块偏向“输入任务”，自动任务模块更偏向“管理任务”。

我对这个模块的定位是：

```text
任务创建入口
→ 任务列表管理
→ 任务状态展示
→ 任务操作控制
```

也就是说，用户可以在这里看到自己创建了哪些任务、任务是否正在进行、是否暂停、是否过期，也可以继续进行查看、暂停、启用、删除等操作。

---

### 2. 页面结构设计

自动任务页面目前主要包括以下结构：

```text
自动任务页面
├── 页面标题：自动任务
├── 说明文字
├── 顶部操作区
│   ├── 搜索框 / 搜索按钮
│   ├── 状态筛选下拉框
│   └── 新建自动任务按钮
│
├── 任务卡片列表
│   ├── 任务名称
│   ├── 任务状态
│   ├── 时间信息
│   ├── 查看按钮
│   └── 更多操作菜单
│
└── 新建任务弹窗
    ├── 名称
    ├── 要求说明
    ├── 执行时间
    ├── 日期选择
    ├── 时间选择
    └── 创建 / 取消按钮
```

这个结构参考了 Marvis 官方自动任务页面的基本布局：顶部有标题和说明，右上方有搜索、筛选和新建按钮，中间是任务卡片列表，点击新建后弹出任务创建窗口。

---

### 3. 前端制作思路

自动任务模块的制作重点不是单纯做一个表单，而是要把“任务管理”的感觉做出来。

首先，我把顶部操作区设计成任务管理的控制中心。搜索、筛选和新建按钮放在同一层级，方便用户快速查找任务、切换任务状态或创建新任务。

其次，任务内容采用卡片式展示。相比表格，卡片更接近 Marvis 的产品风格，也更适合展示任务名称、状态、时间和操作按钮。每个任务卡片都可以理解为一个独立的任务对象，后续可以继续扩展更多字段，例如执行记录、失败原因、下次执行时间等。

最后，新建自动任务采用弹窗方式完成。这样用户不需要离开当前页面，就可以快速创建任务。弹窗中保留了任务名称、任务说明和执行时间几个核心字段，避免一开始就做得过于复杂。

---

## 四、新建对话与自动任务之间的关系

这两个模块虽然是两个页面，但在产品逻辑上其实是连在一起的。

新建对话模块负责接收用户的自然语言输入，自动任务模块负责任务的管理。例如用户在新建对话中输入：

```text
明天上午 9 点提醒我提交作业
```

系统后续应该能够识别出这是一个自动任务，然后把任务写入任务列表中，最终在自动任务页面中展示出来。

因此它们之间的关系可以理解为：

```text
新建对话页面
→ 用户输入自然语言任务
→ 系统判断任务类型
→ 如果是定时任务，则创建自动任务
→ 自动任务页面展示和管理该任务
```

所以我这次做这两个模块，并不是单独完成两个静态页面，而是在搭建一个基础业务闭环：**用户提出任务，系统保存任务，用户可以继续管理任务**。

# Alice(6.18)
# Marvis 复刻项目前端阶段总结：技能广场模块

本次开发中，我继续完成了 Marvis 复刻项目前端部分的第三个核心板块：**技能广场模块**。如果说新建对话模块负责用户输入任务，自动任务模块负责任务管理，那么技能广场模块更像是整个 AI 助手能力的展示和扩展入口。它主要用于展示不同类型的 Skill、推荐内容、工具能力以及技能详情页面，让用户能够通过可视化方式理解 Marvis 可以完成哪些事情。

相比前两个模块，技能广场模块的前端难点不只是做出一个页面，而是要同时处理“内容社区感”和“工具库管理感”。因此这次我重点围绕两个页面方向进行设计：一个是偏内容推荐的“探索发现”，另一个是偏工具管理的“Skill库”。

---

## 一、技能广场模块定位

技能广场模块是 Marvis 项目中用于展示 AI 能力生态的入口。它不是一个普通的功能列表，而是把各种可调用的 Skill、自动化能力、效率工具、内容创作工具集中展示出来，让用户可以通过分类浏览、搜索筛选和详情查看来选择自己需要的能力。

我对这个模块的定位是：

```text
技能展示入口
→ 技能分类浏览
→ 技能搜索与筛选
→ 技能详情查看
→ 后续接入技能安装 / 调用 / 管理能力
```

也就是说，技能广场不仅是一个静态展示页面，后续还可以继续扩展成真正的 Skill 管理中心。用户可以在这里发现新的工具，也可以进入 Skill库查看已经具备或可以使用的技能。

---

## 二、页面结构设计

技能广场模块目前主要包括以下结构：

```text
技能广场页面
├── 顶部切换区
│ ├── 探索发现
│ └── Skill库
│
├── 分类筛选区
│ ├── 全部
│ ├── 办公效率 / 办公学习
│ ├── 电脑设置
│ ├── 生活服务 / 生活日常
│ ├── 内容创作
│ ├── 理财投资
│ └── 学术研究
│
├── 工具操作区
│ ├── 搜索按钮 / 搜索框
│ ├── 排序下拉框
│ └── 我的技能按钮
│
├── 探索发现内容区
│ ├── 推荐技能卡片
│ ├── 分类技能卡片
│ └── 热门内容卡片
│
├── Skill库内容区
│ ├── 工具型 Skill 卡片
│ ├── 技能名称
│ ├── 技能说明
│ ├── 来源信息
│ └── 安装 / 使用量信息
│
└── 技能详情页
 ├── 返回按钮
 ├── 技能封面 / 海报区域
 ├── 技能标题
 ├── 作者与分类信息
 ├── 详细说明
 ├── 相关工具
 ├── 评论输入
 └── 点赞 / 评论信息
```

这个结构主要参考了 Marvis 官方技能广场的整体逻辑：顶部先区分“探索发现”和“Skill库”，中间通过分类标签切换不同内容，右侧提供搜索和排序功能。用户点击某个技能卡片后，可以进入对应的详情页，查看该 Skill 的详细介绍、相关工具和互动信息。

---

## 三、探索发现页面总结

### 1. 模块定位

探索发现页面更偏向内容推荐和社区发现。它的核心作用是让用户快速看到当前热门的 Skill、合集内容、推荐工具和不同分类下的能力展示。

我在设计这个页面时，没有把它做成传统的工具列表，而是更偏向“内容流”的形式。因为 Marvis 官方的技能广场更像一个内容社区，用户看到的不只是工具名称，而是带有封面、标题、作者、点赞和评论的推荐卡片。

因此探索发现页面可以理解为：

```text
内容推荐入口
→ 热门 Skill 展示
→ 分类内容浏览
→ 点击卡片进入详情
```

### 2. 前端制作思路

探索发现页面主要采用卡片式网格布局。每张卡片由封面区域和信息区域组成，封面区域用于展示技能主题，信息区域用于展示标题、作者、点赞数和评论数。

在制作过程中，我重点考虑了三个问题。

第一，卡片需要有统一的视觉风格。为了接近 Marvis 官方的技能广场效果，我采用了大圆角、浅色背景、轻阴影和渐变封面的设计方式，让页面整体看起来更接近桌面端应用，而不是普通网页列表。

第二，分类切换需要对应不同内容。比如办公效率、生活服务、内容创作、理财投资、学术研究等分类下，应该展示不同的卡片内容，而不是所有分类都复用同一批数据。因此我对不同分类准备了不同的 mock 数据，使页面切换时能够体现真实的内容变化。

第三，卡片需要能够进入详情页。技能广场不是只停留在列表展示，点击卡片后应该进入更详细的 Skill 介绍页面，所以我预留了详情页跳转逻辑，用于展示技能封面、介绍文本、相关工具和互动区域。

---

## 四、Skill库页面总结

### 1. 模块定位

Skill库页面更偏向工具管理。它和探索发现不同，探索发现强调“看见和发现”，而 Skill库强调“查找和使用”。用户在这里更关心某个 Skill 能做什么、属于哪个分类、来源是什么，以及后续是否可以添加到自己的技能中。

我对 Skill库页面的定位是：

```text
技能工具列表
→ 已有能力管理
→ 技能快速查找
→ 后续接入技能安装和调用
```

### 2. 页面结构设计

Skill库页面采用更紧凑的工具卡片布局。相比探索发现中的海报式卡片，Skill库中的卡片更像工具条目，每个卡片重点展示图标、Skill 名称、简短说明、来源平台和安装数量。

当前 Skill库页面主要包括：

```text
Skill库页面
├── 顶部分类筛选
├── 搜索与排序按钮
├── 我的技能按钮
└── Skill 工具卡片网格
   ├── 图标
   ├── Skill 名称
   ├── 简短描述
   ├── 来源
   └── 使用 / 安装信息
```

这种结构的好处是信息密度更高，用户可以快速扫视多个工具，而不会像探索发现那样被大面积封面占据过多空间。

---

## 五、技能详情页总结

技能详情页是技能广场模块中比较重要的补充页面。它的作用是承接用户从卡片进入后的详细查看需求。

目前详情页主要采用左右双栏结构：

```text
技能详情页
├── 左侧
│ ├── 返回按钮
│ ├── 技能大封面
│ └── 轮播指示点
│
└── 右侧
 ├── 技能标题
 ├── 作者 / 分类信息
 ├── 技能介绍正文
 ├── 相关工具列表
 ├── 评论输入框
 ├── 点赞数
 └── 评论数
```

左侧主要承担视觉展示作用，右侧负责展示文字信息和工具信息。这样的布局比较接近 Marvis 官方技能详情页的设计逻辑，也能让用户在查看 Skill 时同时关注“这个 Skill 是什么”和“它包含哪些工具”。

---

## 六、技能广场与其他模块之间的关系

技能广场模块虽然是一个独立页面，但它和新建对话、自动任务模块并不是割裂的。

新建对话模块负责自然语言输入，自动任务模块负责任务管理，而技能广场模块则负责展示系统具备的能力。三者之间可以形成这样的关系：

```text
技能广场
→ 用户发现某个 Skill
→ 用户理解该 Skill 可以完成什么任务
→ 用户在新建对话中调用相关能力
→ 如果任务需要定时执行，则进入自动任务模块管理
```

例如用户在技能广场中看到“电脑体检”“本地发票整理”“PPT 一键优化”等技能，后续就可以在新建对话中直接输入对应需求，让系统通过 Agent 调度具体工具完成任务。如果某些 Skill 需要定时运行，也可以和自动任务模块进行连接。

因此，技能广场本质上是整个项目的“能力展示层”。它让用户知道系统有哪些能力，也为后续接入真实工具、插件、MCP、Function Calling 和本地文件能力提供了前端入口。
