# Marvis 复刻项目前端阶段总结：新建对话与自动任务模块

本次开发中，我主要完成了 Marvis 复刻项目前端部分的两个核心板块：**新建对话模块**和**自动任务模块**。这两个模块是整个桌面端 AI 助手的基础入口，一个负责用户与系统的自然语言交互，另一个负责任务的创建、展示和管理。相比单纯堆页面，我这次更关注的是前端整体框架、页面结构、模块之间的关系，以及后续如何继续接入真实的 Agent 能力。

---

## 一、整体前端框架设计思路

这次前端采用的是桌面端应用的设计思路，而不是普通网页页面。整体结构围绕一个主窗口展开，左侧是固定的导航栏，右侧是不同功能页面的内容区。这样的设计比较接近 Marvis 官方客户端的使用体验，也更适合后续扩展为一个本地 AI 助手。

目前前端整体可以理解为：

```text
主窗口
├── 左侧 Sidebar 导航栏
│   ├── 新建对话
│   ├── 自动任务
│   ├── 技能广场
│   ├── 应用
│   ├── 文档
│   ├── 图库
│   ├── 此电脑
│   └── 设置
│
└── 右侧主内容区域
    ├── 新建对话页面
    ├── 自动任务页面
    └── 其他功能页面
```

我这次的重点不是把所有页面都做完整，而是先把最核心的两个功能入口搭出来：**用户从哪里输入任务，以及任务创建后在哪里管理**。因此，“新建对话”和“自动任务”实际上构成了当前项目前端的主流程闭环。

---

## 二、新建对话模块总结

### 1. 模块定位

新建对话模块是整个项目的主入口。它并不是一个简单的聊天页面，而是一个面向 AI 助手的任务输入入口。用户可以在这里通过自然语言输入需求，例如让系统打开应用、查找文件、创建提醒任务，或者执行某个本地操作。

所以我在设计这个模块时，没有把它只当成普通聊天窗口，而是把它定位为：

```text
自然语言输入入口
→ 任务理解入口
→ Agent 调度入口
→ 结果展示入口
```

也就是说，用户在这里输入一句话，后续可以交给 Main Agent 或其他功能模块去判断任务类型，然后再分发给文件、应用、自动任务、系统信息等不同能力。

---

### 2. 页面结构设计

新建对话页面目前主要由以下几部分组成：

```text
新建对话页面
├── 顶部或中央欢迎区域
├── 主输入框
├── 快捷任务提示
├── 执行状态展示
├── 结果展示区域
└── 日志 / 状态反馈区域
```

其中最核心的是输入框和任务反馈区域。输入框负责接收用户的自然语言任务，反馈区域则用于展示执行结果、状态变化或任务日志。

我在设计时尽量避免把页面做成普通聊天软件那种纯消息列表，而是更偏向“AI 操作台”的风格。因为 Marvis 这类产品的重点不是陪聊，而是让用户通过一句话驱动本地任务。

---

### 3. 前端制作思路

新建对话模块的制作思路主要有三点。

第一，我先保证页面结构清晰。左侧导航栏固定，右侧内容区域根据点击切换页面，这样整个项目不会变成零散页面，而是有统一的应用框架。

第二，我把输入区域作为页面视觉中心。因为用户打开软件后，最直接的操作就是输入任务，所以输入框的位置、大小和层级需要比较突出。

第三，我预留了结果展示和状态展示区域。后续无论是接入本地模型、Function Calling、MCP 工具，还是接入文件检索、应用启动、自动任务创建，都可以把结果统一返回到这个页面中展示。

目前新建对话模块已经基本完成了“用户输入任务 → 前端接收 → 展示执行反馈”的基础框架。后续需要继续加强的是多轮对话体验、历史记录、快捷 tag 交互，以及与后端 Agent 调度逻辑的深度连接。

---

## 三、自动任务模块总结

### 1. 模块定位

自动任务模块是本次开发中另一个重点。它的作用是让用户创建、查看和管理定时任务。相比新建对话模块偏向“输入任务”，自动任务模块更偏向“管理任务”。

我对这个模块的定位是：

```text
任务创建入口
→ 任务列表管理
→ 任务状态展示
→ 任务操作控制
```

也就是说，用户可以在这里看到自己创建了哪些任务、任务是否正在进行、是否暂停、是否过期，也可以继续进行查看、暂停、启用、删除等操作。

---

### 2. 页面结构设计

自动任务页面目前主要包括以下结构：

```text
自动任务页面
├── 页面标题：自动任务
├── 说明文字
├── 顶部操作区
│   ├── 搜索框 / 搜索按钮
│   ├── 状态筛选下拉框
│   └── 新建自动任务按钮
│
├── 任务卡片列表
│   ├── 任务名称
│   ├── 任务状态
│   ├── 时间信息
│   ├── 查看按钮
│   └── 更多操作菜单
│
└── 新建任务弹窗
    ├── 名称
    ├── 要求说明
    ├── 执行时间
    ├── 日期选择
    ├── 时间选择
    └── 创建 / 取消按钮
```

这个结构参考了 Marvis 官方自动任务页面的基本布局：顶部有标题和说明，右上方有搜索、筛选和新建按钮，中间是任务卡片列表，点击新建后弹出任务创建窗口。

---

### 3. 前端制作思路

自动任务模块的制作重点不是单纯做一个表单，而是要把“任务管理”的感觉做出来。

首先，我把顶部操作区设计成任务管理的控制中心。搜索、筛选和新建按钮放在同一层级，方便用户快速查找任务、切换任务状态或创建新任务。

其次，任务内容采用卡片式展示。相比表格，卡片更接近 Marvis 的产品风格，也更适合展示任务名称、状态、时间和操作按钮。每个任务卡片都可以理解为一个独立的任务对象，后续可以继续扩展更多字段，例如执行记录、失败原因、下次执行时间等。

最后，新建自动任务采用弹窗方式完成。这样用户不需要离开当前页面，就可以快速创建任务。弹窗中保留了任务名称、任务说明和执行时间几个核心字段，避免一开始就做得过于复杂。

## 四、新建对话与自动任务之间的关系

这两个模块虽然是两个页面，但在产品逻辑上其实是连在一起的。

新建对话模块负责接收用户的自然语言输入，自动任务模块负责任务的管理。例如用户在新建对话中输入：

```text
明天上午 9 点提醒我提交作业
```

系统后续应该能够识别出这是一个自动任务，然后把任务写入任务列表中，最终在自动任务页面中展示出来。

因此它们之间的关系可以理解为：

```text
新建对话页面
→ 用户输入自然语言任务
→ 系统判断任务类型
→ 如果是定时任务，则创建自动任务
→ 自动任务页面展示和管理该任务
```

所以我这次做这两个模块，并不是单独完成两个静态页面，而是在搭建一个基础业务闭环：**用户提出任务，系统保存任务，用户可以继续管理任务**。

## 六、本次开发的收获

这次开发让我更加明确了一点：复刻 Marvis 这类产品，不能只盯着界面像不像，而是要先理解它的产品结构。

新建对话页面的核心不是聊天，而是任务输入；自动任务页面的核心不是表单，而是任务管理。只有先把这两个模块的定位想清楚，后续前端结构才不会混乱。

同时我也意识到，AI 助手类产品的前端并不是简单展示数据，而是需要和后端 Agent 系统形成配合。前端要能表达任务状态、执行过程、用户确认、风险提示和最终结果，这就要求页面设计一开始就要考虑扩展性。

本次完成的新建对话和自动任务模块，为后续继续开发文件管理、应用启动、技能广场、本地知识库、任务调度等功能打下了基础。

---

## 七、后续优化方向

接下来我准备继续从两个方向优化。

第一是继续提升前端还原度。当前页面已经有了基础结构，但在组件间距、按钮样式、弹窗质感、任务卡片层级、搜索框交互等方面，还可以继续向 Marvis 官方效果靠近。

第二是继续完善功能闭环。新建对话页面后续需要真正接入 Agent 任务分发逻辑，自动任务页面也需要继续完善任务执行、暂停、删除、运行记录、失败重试和后台调度能力。

总体来说，本次开发完成的是 Marvis 复刻项目中非常关键的一步：我已经把“用户如何输入任务”和“用户如何管理任务”这两个核心前端模块搭建起来了。后续只要继续围绕这两个入口扩展能力，整个项目就可以逐步从界面原型走向一个真正可用的本地 AI 助手。
